# 型定義の自動生成導入 (plan-10)

## Context

- **現状**: `src/core/models.py`（Pydantic v2）と `dashboard/functions/api/_types.ts` が手書きでミラーされている。`tests/test_contract.py` が正規表現で TS をパースしてフィールド集合を照合する形で整合性を担保している（`test_contract.py:19-61`）。
- **問題**: 手書きミラーは追加・改名のたびに両側を編集する必要があり、漏れを regex テストで事後的に検出する運用。型情報（optional / union / nested）は集合比較では検証できず、将来の機能追加で破綻しやすい。
- **ゴール**: Pydantic を唯一の真実のソース (SSOT) として `_types.ts` 相当を自動生成し、CI でドリフトを検出する。regex パース方式のテストを撤去する。

---

## 対象と非対象

**自動生成対象（Pydantic 由来）:**
- `SourceDef`, `CategoryDef`, `UserSettings`（`test_contract.py` で検証されている3つ）
- これらが参照する enum / Literal も TypeScript ユニオン型として再生成

**自動生成しない（手書きのまま）:**
- `Env`（Cloudflare Workers 固有の `KVNamespace` を含む）
- `ArticleHistoryEntry`（Python 側は `SelectedArticle`+`Article` の合成で直接対応モデルなし）
- 定数 `VALID_SOURCE_TYPES`, `TYPE_LABELS`（UI ラベル用のハンドロール定数）
- `_kv_keys.ts`（値の照合は既存 `test_kv_keys_match_ts` がカバー、範囲外）

---

## 設計方針

### 採用: `pydantic2ts`（pip）+ `json-schema-to-typescript`（npm）

Pydantic v2 対応の成熟ツール。`pydantic2ts` が Python から `model_json_schema()` → JSON Schema → `json2ts` (npm) に渡して TS を生成する。

**比較:**

| 候補 | 長所 | 短所 | 採否 |
|---|---|---|---|
| `pydantic2ts` | Pydantic v2 対応、JSON Schema 経由で型・union・optional を正確に再現 | Python と Node の両方が必要 | **採用** |
| `datamodel-code-generator` | 成熟 | 方向が逆（JSON Schema → Pydantic）。本件では使えない | 不採用 |
| OpenAPI (FastAPI) | 標準化された仕様 | FastAPI 依存の追加はオーバースペック | 不採用 |
| 自作 Python スクリプト（`model_json_schema()` → 手書き TS emitter） | 依存ゼロ | union / nullable / nested の取りこぼし対応を自前で書く必要あり | 不採用 |

開発環境には既に Python と Node が揃っている（`dashboard/package.json` で Node、`pyproject.toml` で Python）ので二重要件は問題にならない。

### 出力ファイル構成

`_types.ts` の import 元は複数ファイル（`settings.ts`, `articles.ts`, `stats.ts`, `page.tsx`, `SourceEditor.tsx` 等）に散らばっているため、**既存 import を壊さない構成**を取る:

```
dashboard/functions/api/
├── _types.generated.ts  ← 自動生成（触らない／手編集禁止のヘッダ付）
└── _types.ts            ← 手書き: Env, ArticleHistoryEntry, 定数
                           + re-export: SourceDef, CategoryDef, UserSettings from ./_types.generated
```

消費側（`import type { SourceDef } from "./_types"` など）は無変更で動く。

### Step 2: 生成スクリプト `scripts/gen_types.py` を作成

```python
"""Pydantic モデル → dashboard 用 TypeScript 型定義生成。

出力: dashboard/functions/api/_types.generated.ts
対象: SourceDef, CategoryDef, UserSettings
"""
from pathlib import Path
from pydantic2ts import generate_typescript_defs

ROOT = Path(__file__).resolve().parent.parent
OUT = ROOT / "dashboard" / "functions" / "api" / "_types.generated.ts"

def main() -> None:
    generate_typescript_defs(
        "src.core.models",
        str(OUT),
        json2ts_cmd=str(ROOT / "dashboard" / "node_modules" / ".bin" / "json2ts"),
    )
    # pydantic2ts は全モデルを出力するので、不要モデル（Article, SelectedArticle,
    # ArticleFeedback, UserPreferences）を削除 or whitelist 方式で取捨。
    # 実装では generate_typescript_defs の代わりに model_json_schema() を直接呼び、
    # 必要モデルだけを渡すラッパ関数にする（後述）。

if __name__ == "__main__":
    main()
```

- [ ] `pydantic2ts` は指定モジュール内の **全** `BaseModel` を拾うため、対象 3 モデルのみに絞り込む小さなラッパを書く（`model_json_schema()` の `$defs` を使って結合スキーマを作り一括で渡す方式、あるいは `exclude` 引数を使う方式のどちらか、ツールの現仕様確認後に決定）
- [ ] 生成ファイル先頭に `// AUTO-GENERATED by scripts/gen_types.py — DO NOT EDIT` を挿入
